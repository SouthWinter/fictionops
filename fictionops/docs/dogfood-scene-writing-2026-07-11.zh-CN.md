# 逐场景真实模型 Dogfood（2026-07-11）

## 范围

使用 OpenAI-compatible DeepSeek runner，在一个不含目标原稿的最小长篇项目中测试高风险物件状态章节。输入只包含章节发动机、相邻状态摘要、人物/物件规则、风格约束和作者显式偏好。

正式稿未被模型读取或修改，所有输出停留在 run 目录。

## 结果

- 最终工作流完整走过 causal、planner、逐场景 writer、逐场景 rewriter、adversarial reviewer 和 evaluator。
- 最终状态为 `needs_revision_attention`，没有开放采纳。
- 初稿约 1.19 万非空白字符；逐场复修稿约 1.13 万字符。
- 事实账本、对象状态码、事件映射和确定性视角检查通过。
- 候选仍存在主题代答、动作功能重复、显式对照句过密、段落过长和神秘因果过亮等文学问题。
- 一次完整运行使用 21 次模型调用，约 12 分钟。逐场景稳定性显著提高，但成本需要 controller 预算约束。

## Dogfood 驱动的实现变化

- causal/planner/reviewer/evaluator 的有界 JSON/schema 修复；
- 写作字数等元数据与故事事实账本隔离；
- 物件状态使用稳定 code、transition id 和 event id；
- planner 的机器状态码可从 assertions 确定性规范，语义冲突仍阻断；
- planner 场景目标按比例归一到章节虚目标；
- 每场最多两次欠长/超长修复；
- 逐场景复修替代失败后的整章重写；
- 外部可见动作不再被确定性规则误判为视角泄漏；
- reviewer/evaluator 的失败证据必须落在候选原文中。

## 结论

逐场景生成解决了通用 API 模型整章输出过短与局部状态遗忘问题，但没有自动解决文学质量。控制器现在能够保存状态、修复结构、维持体量、组织复修并拒绝不合格候选；“写得比正式作者稿好”仍未被证明，不能把工作流跑通等同于质量达标。

## 运行后加固

v10 的 21 次调用说明“所有场景一起复修”成本过高，也会让无问题场景承受不必要的文风漂移。运行后新增两项 controller 能力：

- reviewer 原文证据和失败 `scene_id` 会路由到具体场景，只复修命中的场景；无法定位或全章静态失败时才全场复修；
- 写章与修订工作流新增调用次数/总时限硬预算，每次 runner 调用前检查，并持久化 `model_budget.json`。

这些变化已有确定性回归测试，但没有倒写进 v10 的调用统计，也尚未再花真实 API 成本重跑同一章。

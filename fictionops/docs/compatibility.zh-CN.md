# FictionOps 兼容性策略

FictionOps 的文件和 JSON 报告会被人类作者、脚本和外部 controller 读取。因此，即使还没到 1.0，兼容性也是产品的一部分。这份文档定义当前哪些东西可以依赖，以及行为变化时必须怎样记录。

## 版本阶段

| 版本阶段 | 兼容性承诺 |
| --- | --- |
| `0.x` pre-alpha | 命令仍可能演进，但已有测试保护的行为不应静默破坏。破坏性变化必须写入 `CHANGELOG.md`、CLI 契约和发布说明。 |
| `1.x` stable core | 命令名、主要文件位置和核心 JSON key 应在同一 major 版本内保持向后兼容。破坏性变化需要新的 major 版本或明确迁移路径。 |

## 稳定面

这些部分对兼容性敏感：

- CLI 命令名和必填参数。
- 退出码约定：成功为 `0`，失败为非零，并在 stderr 给出可操作错误。
- 默认读写安全：审计命令默认只读，除非传入 `--out`；写文件命令默认拒绝意外覆盖，除非使用 `--force` 或很窄的安全替换选项。
- dataclass 报告里的核心 JSON key。
- 写入持久文件的 schema 字符串，例如 agent run request、agent exec receipt、publish manifest 和 metadata export。
- `init`、`new-book`、`new-chapter` 生成的标准项目目录。
- Agent 暂存契约：`agent-run` 准备有边界输入，`agent-exec` 写 staging 输出，`agent-inbox` 检查 staging 输出，FictionOps 不自动应用模型输出。

## 兼容的新增

这些变化通常是兼容的：

- 增加带安全默认值的可选 CLI 参数。
- 保留旧 key 含义的前提下增加 JSON key。
- 给人读的 Markdown 报告增加新章节。
- 增加低优先级提示类审计项。
- 增加示例、文档、模板或报告元数据。

外部 controller 应忽略未知 JSON key，并依赖命名字段而不是字段顺序。

## 破坏性变化

这些变化属于破坏性变化，必须明确记录：

- 删除或重命名 CLI 命令。
- 改变必填参数，或以会移动用户数据的方式改变默认路径。
- 删除 JSON key 或改变 key 含义。
- 改变 schema 字符串但不给迁移路径。
- 让覆盖行为变得不保守。
- 让模型输出直接修改正文或正史文件。
- 把原本不安全的 controller 步骤改成可自动运行，且该步骤会影响正文、正史、凭据或不可逆发布物。

## 废弃路径

当行为必须变化时：

1. 尽量先新增替代行为，不立刻删除旧行为。
2. 在 `CHANGELOG.md` 和相关 CLI 文档中标记旧行为已废弃。
3. 过渡期同时测试旧行为和新行为。
4. 如果必须移除，更新发布说明并提供迁移说明。
5. 除非用户显式选择，否则默认保留更安全的行为。

## Controller 指南

外部 agent 和 controller 应该：

- 使用 `--format json` 获取机器可读输出。
- 行动前检查 `status`、`ready`、`issue_count`、`blocking_issue_count` 和命令特有安全字段。
- 遇到未知 status 时停在人类复核边界。
- 把未知 JSON key 当成无害新增。
- 如果预期 key 缺失，视为不兼容并停止。
- 有 JSON 输出时，不要解析 Markdown 表格。
- 不要把 `agent-next` 建议理解成可以应用模型暂存输出的授权。

## 文件兼容

FictionOps 文件是普通 Markdown、YAML、JSON 和 EPUB。项目不拥有用户的写作内容，兼容性策略必须优先保护用户手写内容。

规则：

- 生成模板可以演进，但已有项目文件不会自动重写。
- 命令应保留非空用户字段，除非用户显式使用并理解 `--force`。
- 迁移工具应先复制到沙盒，再移动导入正文。
- 发布物可以重新生成，但源草稿和正史文件保持分离。

## 发布证据

发布候选应通过这些证据证明兼容性敏感行为：

- 每个 MVP 命令都有 CLI help 覆盖。
- 写入/拒绝覆盖行为有回归测试。
- JSON 输出有 `json.loads` 解析测试。
- 包内容测试覆盖源码、模板、文档、示例和测试。
- 使用构建包进行安装烟测。
- 当兼容性边界变化时，发布说明和已知限制也同步更新。

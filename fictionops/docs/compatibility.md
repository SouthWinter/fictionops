# FictionOps Compatibility Policy

FictionOps is designed for humans and external controllers to read its files and JSON reports. That means compatibility is part of the product, even before 1.0. This policy defines what users can rely on today and what must be documented when the project changes.

## Status By Version

| Version Stage | Compatibility Promise |
| --- | --- |
| `0.x` pre-alpha | Commands may still evolve, but existing tested behavior should not break silently. Breaking changes must be documented in `CHANGELOG.md`, CLI contracts, and release notes. |
| `1.x` stable core | Command names, major file locations, and core JSON keys are expected to remain backward compatible within the major version. Breaking changes require a new major version or an explicit migration path. |

## Stable Surfaces

These surfaces are treated as compatibility-sensitive:

- CLI command names and required arguments.
- Exit-code convention: success is `0`, failures are non-zero with actionable stderr.
- Default read/write safety: audit commands are read-only unless `--out` is passed; write commands avoid accidental overwrites unless `--force` or a narrow safe-replacement flag is used.
- Core JSON keys in dataclass-backed reports.
- Schema strings written to durable files, such as agent run requests, agent exec receipts, publish manifests, and metadata exports.
- Standard project folders generated by `init`, `new-book`, and `new-chapter`.
- Agent staging contract: `agent-run` prepares bounded inputs, `agent-exec` writes staged output, `agent-inbox` reviews staged output, and FictionOps never applies model output automatically.

## Additive Changes

These changes are normally compatible:

- Adding optional CLI flags with safe defaults.
- Adding new JSON keys while preserving existing key meanings.
- Adding new Markdown sections to human-readable reports.
- Adding new advisory checks that produce lower-priority notes.
- Adding examples, docs, templates, or report metadata.

External controllers should ignore unknown JSON keys and prefer named keys over positional assumptions.

## Breaking Changes

These changes are breaking and require explicit documentation:

- Removing or renaming a CLI command.
- Changing a required argument or default path in a way that moves user data.
- Removing a JSON key or changing its meaning.
- Changing a schema string without providing a migration path.
- Changing overwrite behavior to be less conservative.
- Letting model output directly modify manuscript or canon files.
- Reclassifying a previously safe controller step as auto-runnable when it can affect manuscript, canon, credentials, or irreversible publishing artifacts.

## Deprecation Path

When a behavior needs to change:

1. Add the replacement behavior without removing the old behavior when possible.
2. Document the old behavior as deprecated in `CHANGELOG.md` and the relevant CLI docs.
3. Add tests for both old and new behavior during the transition.
4. If removal is unavoidable, update release notes and provide a migration note.
5. Keep the safer behavior as the default unless the user explicitly opts in.

## Controller Guidance

External agents and controllers should:

- Use `--format json` for machine-readable output.
- Check `status`, `ready`, `issue_count`, `blocking_issue_count`, and command-specific safety fields before acting.
- Treat unknown statuses as human-review boundaries.
- Treat unknown JSON keys as harmless additions.
- Treat missing expected keys as incompatible and stop.
- Never parse Markdown tables when JSON output exists.
- Never assume `agent-next` recommendations are permission to apply staged model output.

## File Compatibility

FictionOps files are ordinary Markdown, YAML, JSON, and EPUB files. The project does not own the user's writing, and compatibility policy must favor preserving user-authored content.

Rules:

- Generated templates may evolve, but existing project files are not rewritten automatically.
- Commands should preserve non-empty user fields unless `--force` is explicit and documented.
- Migration tools should copy into a sandbox before mutating imported draft locations.
- Publish artifacts may be regenerated, but source drafts and canon files remain separate.

## Release Evidence

A release candidate should prove compatibility-sensitive behavior through:

- CLI help coverage for every MVP command.
- Regression tests for write/no-overwrite behavior.
- JSON parsing tests with `json.loads`.
- Package content tests for source, templates, docs, examples, and tests.
- Installation smoke tests using the built package.
- Release notes and known-limits updates when compatibility boundaries change.
- Recovery playbook updates when a new command can create, repair, regenerate, or invalidate durable project state.

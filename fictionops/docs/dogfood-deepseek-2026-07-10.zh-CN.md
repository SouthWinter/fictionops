# DeepSeek 真实写作 Dogfood（2026-07-10）

## 目标

用真实 DeepSeek API 验证两条核心链路：

1. 对旧章进行全面审读与修订；
2. 仅给写新章的必要条件，独立起草一章。

两次运行都使用 OpenAI-compatible runner 调用 `deepseek-chat`，所有候选稿留在 staging 目录，没有覆盖正文。

## 旧章修订实验

- 目标：第一本第 26 章。
- 审读能正确识别新的比喻族规则，没有要求机械轮换“像/好像/仿佛”。
- 候选稿降低了部分“不是”和“忽然”，但“冷”的计数没有下降，并且凭空新增记忆片段。
- 语义 verifier 仍声称高优先级问题已解决，导致假阳性 `ready_for_approval`。

结论：修订验收不能只依赖模型自证。P1/P2 中声明必须下降的可量化问题，现在必须与 static ledger 的 `metric_deltas` 一致，否则阻断采纳。

## 新章起草实验

- 目标：第二本第 5 章《开仓雨》。
- 上下文：4 个文件，约 2056 字符。
- 仅包含：章节发动机、局部书纲、第 4 章余波、人物/知识边界与最小风格约束。
- 沙盒中不含已写的第 5 章或第 6 章正文。
- 候选稿约 8304 个非空白字符，与正式第 5 章没有 12 字符长的共享 n-gram，可视为从最小条件独立生成。
- 候选稿仍违背了明确约束：让鹿煜提前形成成熟抽象理论，直接进入承曦视角，并以多段主题总结收尾。
- 模型 evaluator 却逐项声称约束已遵守，再次产生假阳性。

结论：约 2000 字符的高密度必要条件已足以支撑一次完整起草，但不足以让同一模型可靠地自我验收。新的 evaluator 协议因此要求：

1. 为每条 preserve/forbidden 约束分配稳定 ID；
2. 逐条尝试证伪，并引用候选稿证据；
3. 任一约束 `fail` 或 `uncertain` 时阻断采纳；
4. 宽泛的维度评分不得覆盖具体约束失败。

## 工程含义

这次 dogfood 证明了两件事：

- Context Compiler 的最小上下文路线可行，不需要把整本书塞给模型；
- Verifier 必须是“证据门禁 + 模型判断”，不能只是第二次模型自评。

本次两份候选稿都未被采纳，它们是失败证据，不是可发布正文。

## Memory-first Agent v1 追加实验

新闭环在同一最小沙箱上调用真实 `deepseek-chat`：

- 记忆层检索 11 条相关记忆；
- causal simulator 正确把视角白名单限定为辛照渠/鹿煜，将承曦列为禁止视角；
- causal simulator 将“百姓是否是整齐称呼”保留为 theme question；
- planner 却用“第三人称有限”包装承曦场景，并在鹿煜出口知识中直接回答主题；
- 计划契约门禁稳定识别了两类违规，并要求只修计划；
- DeepSeek 第一次只删除承曦视角，第二次把主题答案从 exit state 移到 goal，证明了“移位合规”风险；
- controller 因此将主题答案检查扩展到全部场景字段，并在因果模拟后为 planner/writer 生成不含现成答案的执行上下文；
- 候选稿只有 4838 个非空白字符，未达 8000 下限，同时硬审计另外命中了承曦视角和主题直答，因此正确停在 `needs_revision_attention`。

前两次真实连接遇到 SSL EOF。Chat runner 随后增加了仅针对传输错误、HTTP 429 和 5xx 的有限指数退避重试；鉴权和其他 4xx 仍立即失败。第三次运行成功完成真实 API 调用。

追加实验再次确认：模型会尝试在字段之间移动同一答案，所以 Agent 必须验证状态和语义契约，而不是只检查某个关键字段。

# FictionOps 审计优先级

> 目标：让长篇修订有先后，不被“所有问题都值得修”拖进无穷返工。

## 1. 核心原则

修订优先级不是按“我更喜欢哪种写法”排序，而是按问题对长篇系统的破坏程度排序。

一句话规则：

**先修会破坏故事承诺的问题，再修会破坏阅读压力的问题，最后修会破坏文字新鲜感的问题。**

## 2. 优先级分层

### P0：正史崩坏

必须立刻修。否则后续内容无法可靠继续。

典型问题：

- 时间线互相矛盾；
- 已死人物再次无解释出现；
- 物件位置不可能；
- 角色掌握了没有来源的信息；
- 已废弃设定进入正文；
- 当前大纲与已写正文发生硬冲突。

处理方式：

1. 先判断正文优先还是大纲优先。
2. 同步 `05_canon`、书纲和相关章节。
3. 在决策记录中写明为何保留这个版本。

### P1：信息边界泄露

优先修。它会让悬念、政治、神话和人物判断失去层次。

典型问题：

- 角色像读过作者设定集；
- 秘密在不安全场合被公开讨论；
- 公共谣言扩散速度超过世界物流；
- 大多数人提前知道本该少数人知道的规则；
- 作者真相被旁白提前盖章。

处理方式：

1. 回到信息释放表。
2. 确认“谁知道、怎么知道、以什么版本知道”。
3. 把可靠解释改成误读、传闻、沉默、局部证据或动作反应。

### P2：人物弧线与智慧模式失真

优先修。它会让角色变成工具人或同一个聪明人的不同名字。

典型问题：

- 少年说话像成熟谋士；
- 底层人物使用朝堂式抽象表达；
- 小妾、商人、军人、祭司的观察方式趋同；
- 角色没有符合自身盲点的错误；
- 情绪后果被过快消解；
- 人物状态在下一章被重置。

处理方式：

1. 回到人物弧线和智慧模式表。
2. 保留人物自己的错误、局限和看不见。
3. 必要时减少戏份，改为动作、旁听、误解或沉默。
4. 运行 `fictionops audit-characters`，确认人物索引、人物弧线、智慧模式和口吻资料没有维护缺口。

### P3：章节发动机不足

需要修。它会让章节像“把大纲点写完”，但读者没有被局势推着走。

典型问题：

- 本章只有信息，没有压力；
- 视角人物没有具体欲望；
- 阻碍太弱，选择没有代价；
- 章末没有不可回退变化；
- 每章都靠揭晓或解释收束；
- 章节体量靠补字数撑起。

处理方式：

1. 重写五列发动机。
2. 给视角人物一个当下欲望。
3. 增加阻碍、误判或代价。
4. 把解释改成场景推进。

### P4：风格与读者体验问题

可批量修。它通常不影响正史，但会影响读者的呼吸、信任和新鲜感。

典型问题：

- “不是/没有/有人”等句式过密；
- 排比推着读者理解；
- 章节开头和结尾太同款；
- 高频词造成 AI 模板感；
- 旁白解释太满；
- 对话节奏过于整齐。

处理方式：

1. 先判断这个模式是否在承担功能。
2. 如果功能必要，只减少密度，不清空。
3. 用动作、物件、错听、停顿、场景中断替换解释。

### P5：润色偏好

可最后处理，也可以不处理。

典型问题：

- 某句可以更漂亮；
- 某个词可以更准；
- 某处节奏略可调整；
- 标题有更好方案；
- 个别段落气质可再磨。

处理方式：

- 不要在结构未定前大量润色。
- 发布前集中处理。
- 不因为一个漂亮句子保留错误结构。

## 3. 修订顺序

默认顺序：

1. P0 正史崩坏；
2. P1 信息边界泄露；
3. P2 人物弧线失真；
4. P3 章节发动机不足；
5. P4 风格与读者体验；
6. P5 润色偏好。

如果资源有限，只修前三类，也能保住长篇骨架。

## 4. 什么时候允许先修风格

以下情况可以提前做 P4：

- 某种句式已经明显影响读者信任；
- AI 模板感正在扩散到多章；
- 排比和解释让章节主题被削弱；
- 章节开头/结尾重复导致连载阅读疲劳；
- 即将发布，需要清稿。

但即便如此，也不要让风格修订覆盖正史修订。

## 5. 审计报告格式

```markdown

### P0 正史崩坏
- 无 / 有：

### P1 信息边界
- 问题：
- 建议：

### P2 人物弧线
- 问题：
- 建议：

### P3 章节发动机
- 问题：
- 建议：

### P4 风格与读者体验
- 问题：
- 建议：

### P5 润色
- 可选：
```

## 6. 对 Agent 的约束

Agent 做审计时必须：

1. 先列高优先级问题，不要先夸文字。
2. 不把个人偏好伪装成结构问题。
3. 每个问题尽量指向文件、章节或段落。
4. 区分“必须改”“建议改”“可以保留”。
5. 不为制造工作量而扩大修订范围。
